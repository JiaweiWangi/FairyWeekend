## 0. 设计原则（参考 CrewAI 最佳实践）

### 0.1 Agent 数量决策

> **默认一个 Agent。** 只有当任务需要不同工具、不同人格、或不同 LLM 时才拆分。

| 步骤 | 工具 | 人格 | 能否合并 |
|------|------|------|----------|
| 分析人设 → 决定 POI 类型 | 无（纯推理） | 城市规划师 | ✅ |
| 搜索 POI | 高德 API | 同上 | ✅ |
| 验证 POI | 无（规则检查） | 同上 | ✅ |
| 生成路线 | LLM | 故事生成引擎 | ❌ 不同人格 |

**结论：两个 Agent**

1. **`poiPlanner`** - 分析人设 + 搜索 POI（工具密集型）
2. **`storyGenerator`** - 生成叙事路线（创意型，无工具）

### 0.2 80/20 规则

> 80% 精力在任务设计，20% 在 Agent 设计。

**重点投入：**
- Task 的 description 和 expected_output
- 结构化输出 Schema（Zod）
- 工具的描述和输入定义

## 3. Agent 1: POI 规划师

### 3.1 职责

分析人设卡，决定搜索什么类型的地点，收集候选 POI。

### 3.2 实现方式

- **框架**: `createReactAgent` (ReAct 模式)
- **工具**: `search_poi`, `get_player_profile`, `reverse_geocode`
- **特点**: 一次调用，批量搜索多个关键词

### 3.3 工具调用优化

```typescript
// 优化前：多次调用（已废弃）
search_poi(keyword: "植物园")
search_poi(keyword: "花店")
search_poi(keyword: "独立书店")

// 优化后：一次调用（当前实现）
search_poi(keywords: ["植物园", "花店", "独立书店", "艺术空间", "小众公园"])
```

**优势**：
- 减少 Agent 循环次数（从 4 次降到 1 次）
- 降低 LLM token 消耗
- 提升响应速度约 4 倍

### 3.4 System Prompt

```typescript
const POI_PLANNER_PROMPT = `你是一位城市探索规划师。

## 目标
找到 15 个左右适合的城市场景候选地点。

## 专业背景
- 深谙城市肌理，擅长从人的情绪和身份出发找到有故事的角落
- 你知道：植物学家看咖啡馆和普通人看咖啡馆是完全不同的
- 偏爱有故事感的日常空间，避开网红打卡点

## 工作流程（必须严格遵循）
1. 分析人设的 identity、mood、mission，推断适合的地点类型
2. 考虑玩家的历史偏好（如果有）
3. 根据稀有度调整地点隐秘程度
4. 调用 search_poi 工具，一次传入 3-5 个关键词数组
5. 完成搜索后，直接回复用户，列出候选地点

## 停止条件
- 调用 search_poi 工具 1 次即可（传入多个关键词）
- 收集到足够的地点信息后直接给出结果
- 不要多次调用工具，不要继续分析或追问`;
```

### 3.5 输出格式

```typescript
interface POIPlannerOutput {
  keywords: string[];      // 搜索的关键词
  reasoning: string;       // 决策理由
  candidates: POI[];       // 候选 POI 列表（约 15-20 个）
}
```

## 4. Agent 2: 故事生成师

### 4.1 职责

基于候选 POI，生成完整的叙事路线。

### 4.2 实现方式

- **框架**: 纯 LLM + `withStructuredOutput`
- **工具**: 无（输入已由 Agent 1 准备好）
- **温度**: 0.7（创意任务）

### 4.3 System Prompt

```typescript
const STORY_GENERATOR_PROMPT = `你是「今日人设」的故事引擎。

你的目标是：将真实地点编织成一段有情绪弧线的城市故事。

你的背景：
- 擅长用第二人称叙事
- 知道好的周末不是打卡清单，而是一段有起承转合的故事
- 文字克制但有温度，像朋友在耳边轻声建议
- 总是给具体行动指引，不模糊说"感受氛围"

规则：
1. 从候选地点中选择 3-4 个，地理上能走通
2. 每个场景从人设的第一视角叙事
3. scene_name 是诗意命名（6-10字）
4. action_task 必须具体可执行
5. 整条路线要有情绪弧线

请以 JSON 格式输出结果。`;
```

### 4.4 输出格式

```typescript
interface Journey {
  story_opening: string;     // 故事开篇（30-50字）
  emotion_arc: {
    start: string;           // 起始情绪
    end: string;             // 结束情绪
  };
  scenes: JourneyScene[];    // 3-4 个场景
  closing: string;           // 故事结语（60-100字）
}

interface JourneyScene {
  order: number;
  scene_name: string;        // 诗意命名
  location_name: string;
  location_type: string;
  location_hint: string;
  persona_narrative: string; // 人设视角叙事
  action_task: string;       // 具体行动
  stay_minutes: number;
  emotion_tags: string[];
  meituan_keyword: string;
}
```

## 5. 工具详解

### 5.1 search_poi

批量搜索高德 POI，支持一次传入多个关键词。

```typescript
// 输入
{
  keywords: ["独立咖啡馆", "小众公园", "独立书店", "花店", "艺术空间"],
  radius: 3000,      // 搜索半径（米）
  lng: 121.47,       // GCJ02 经度
  lat: 31.23         // GCJ02 纬度
}

// 输出
{
  pois: [
    { name, type, address, location, distance },
    // 每个 keyword 最多返回 8 个，去重后合并
  ],
  searchedKeywords: ["独立咖啡馆", "小众公园", ...],
  errors: []  // 可选，记录失败的 keyword
}
```

**限流**: 每次高德 API 调用间隔 500ms（2 QPS）

**计算**: 5 个关键词 × 8 个/关键词 = 最多 40 个 → 去重后约 20-30 个

### 5.2 get_player_profile

从 `dm_memory` 表获取玩家画像。

```typescript
// 输入
{ playerKey: "user_123" }

// 输出
{
  profile: "城市探索者",
  loved_tags: ["咖啡", "书店"],
  disliked_tags: ["网红"],
  visited_pois: ["POI_ID_1", "POI_ID_2"]
}
```

### 5.3 reverse_geocode

WGS84 坐标转 GCJ02，并解析城市名。

```typescript
// 输入
{ lng: 121.47, lat: 31.23 }

// 输出
{
  gcjLng: 121.4737,
  gcjLat: 31.2304,
  city: "上海市"
}
```

## 6. LangGraph 状态与流程

### 6.1 State 定义

```typescript
const QuestState = Annotation.Root({
  // 输入
  card: Annotation<PersonaCard>,
  city: Annotation<string>,
  lat: Annotation<number | undefined>,
  lng: Annotation<number | undefined>,
  playerKey: Annotation<string | undefined>,
  timePeriod: Annotation<string>,      // 上午/下午/傍晚/晚上
  companion: Annotation<string>,       // 独行/情侣/朋友/家人

  // 中间状态
  playerProfile: Annotation<PlayerProfile | undefined>,
  gcjCoords: Annotation<{ lng, lat } | undefined>,
  poiKeywords: Annotation<string[]>,
  poiCandidates: Annotation<POI[]>,

  // 输出
  journey: Annotation<Journey | undefined>,
  error: Annotation<string | undefined>,
});
```

### 6.2 Graph 构建

```typescript
const workflow = new StateGraph(QuestState)
  // 添加节点
  .addNode("fetch_profile", fetchProfile)
  .addNode("resolve_location", resolveLocation)
  .addNode("plan_pois", planPois)
  .addNode("validate_pois", validatePois)
  .addNode("generate_journey", generateJourney)

  // 并行入口
  .addEdge(START, "fetch_profile")
  .addEdge(START, "resolve_location")

  // 汇合点
  .addEdge("fetch_profile", "plan_pois")
  .addEdge("resolve_location", "plan_pois")

  // 顺序流程
  .addEdge("plan_pois", "validate_pois")
  .addEdge("validate_pois", "generate_journey")
  .addEdge("generate_journey", END);
```

### 6.3 验证逻辑

```typescript
// validate_pois 节点
if (poiCandidates.length < 10) {
  warnings.push(`候选不足 (${poiCandidates.length}/10)`);
}

if (types.size < 2) {
  warnings.push(`类型单一 (${types.size}/2)`);
```

## 7. 错误处理

### 7.1 安全检查

所有 Node 函数都有安全检查：

```typescript
// nodes.ts 示例
const keywords = result?.keywords || [];
const candidates = result?.candidates || [];
const scenes = journey.scenes || [];
```

### 7.2 工具失败处理

工具内部 catch，返回空结果而非抛异常：

```typescript
// search-poi.tool.ts
try {
  // ... 调用高德 API
} catch (e) {
  console.warn("search_poi failed:", keyword, e);
  return JSON.stringify({ pois: [], error: String(e) });
}
```

### 7.3 Agent 失败处理

Agent 执行失败时返回空结果，不中断整个流程：

```typescript
// poi-planner.agent.ts
catch (e) {
  return {
    keywords: [],
    reasoning: "Agent 执行失败",
    candidates: [],
  };
}
```
