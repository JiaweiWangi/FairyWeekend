## 0. 设计原则（参考 CrewAI 最佳实践）

### 0.1 Agent 数量决策

> **默认一个 Agent。** 只有当任务需要不同工具、不同人格、或不同 LLM 时才拆分。

| 步骤 | 工具 | 人格 | 能否合并 |
|------|------|------|----------|
| 分析人设 → 决定 POI 类型 | 无（纯推理） | 城市规划师 | ✅ |
| 搜索 POI | 高德 API | 同上 | ✅ |
| 验证 POI | 无（规则检查） | 同上 | ✅ |
| 生成路线 | LLM | 故事生成引擎 | ❌ 不同人格 |

**结论：两个 Agent**

1. **`poiPlanner`** - 分析人设 + 搜索 POI（工具密集型）
2. **`storyGenerator`** - 生成叙事路线（创意型，无工具）

### 0.2 80/20 规则

> 80% 精力在任务设计，20% 在 Agent 设计。

**重点投入：**
- Task 的 description 和 expected_output
- 结构化输出 Schema（Pydantic/Zod）
- 工具的描述和输入定义

## 1. 现状分析

### 1.1 当前流程

```
┌─────────────────────────────────────────────────────────────┐
│                    generate-quest (现有)                      │
├─────────────────────────────────────────────────────────────┤
│  输入: card, city, lat/lng, time_period, companion          │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  1. 坐标转换 (WGS84 → GCJ02)                                 │
│     └─ 调用高德 coordinate/convert API                        │
│                                                              │
│  2. 逆地理编码                                                │
│     └─ 调用高德 geocode/regeo API → 解析城市名称               │
│                                                              │
│  3. POI 关键词映射 (硬编码)  ⚠️ 问题点                         │
│     └─ pickKeywords(identity, mood)                           │
│     └─ 正则匹配 → 返回最多8种类型                             │
│     └─ 例: "植物" → ["公园", "花店", "植物园", "咖啡"...]       │
│                                                              │
│  4. POI 搜索 (批量)                                          │
│     └─ 对每个关键词调用高德 place/around 或 place/text         │
│     └─ 收集最多20个候选                                       │
│                                                              │
│  5. AI 生成路线 (单次调用)                                    │
│     └─ 调用 Lovable AI Gateway (Gemini 2.5 Flash)            │
│     └─ JSON Schema 强制输出结构                               │
│     └─ 输出: story_opening + scenes + closing                │
│                                                              │
├─────────────────────────────────────────────────────────────┤
│  输出: { journey, city, poi_count }                          │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 现有问题

| 问题 | 描述 | 影响 |
|------|------|------|
| **POI 映射硬编码** | `pickKeywords` 用正则匹配，只有约10种模式 | 无法覆盖新人设类型 |
| **无个性化** | 不使用玩家历史画像 | 老玩家每次都是"盲选" |
| **无验证环节** | AI 生成后直接返回，不检查合理性 | 可能生成无效路线 |
| **扩展性差** | 加新功能需要改核心代码 | 维护成本高 |

## 2. Agent 设计目标

### 2.1 核心目标

1. **智能 POI 选择** - LLM 根据人设 + 玩家画像动态决定搜索什么
2. **个性化路线** - 利用 `dm_memory` 中的偏好标签
3. **可扩展** - 新增能力只需加 Tool，不改核心流程
4. **可观测** - 使用 LangSmith 追踪 Agent 决策过程

### 2.2 非目标（本次不改）

- 不改变 API 接口签名（保持向后兼容）
- 不改变数据库结构
- 不改变前端调用方式

## 3. Agent 设计（TypeScript + LangChain.js）

### 3.1 Agent 1: `poiPlanner`（使用 createReactAgent）

**职责**：分析人设卡，决定搜索什么类型的地点，并收集候选 POI

```typescript
import { createReactAgent } from "@langchain/langgraph/prebuilt";
import { ChatOpenAI } from "@langchain/openai";

// LLM 配置（支持 OpenAI 协议的模型）
const llm = new ChatOpenAI({
  model: "gpt-4o-mini",  // 或其他兼容 OpenAI 协议的模型
  temperature: 0.3,
  // 如需自定义 API 地址：
  // configuration: { baseURL: "https://your-api-endpoint/v1" },
});

// System Prompt
const POI_PLANNER_PROMPT = `你是一位城市探索规划师。

你的目标是：根据用户的人设卡和偏好，找到 15-25 个适合的城市场景候选地点。

你的背景：
- 深谙城市肌理，擅长从人的情绪和身份出发找到有故事的角落
- 你知道：植物学家看咖啡馆和普通人看咖啡馆是完全不同的
- 偏爱有故事感的日常空间，避开网红打卡点

规则：
1. 从人设的 identity、mood、mission 出发推断适合的地点类型
2. 考虑玩家的历史偏好（如果有）
3. 稀有度越高，地点越隐秘、越反直觉
4. 使用 search_poi 工具搜索，可以多次调用不同关键词
5. 最终返回 15-25 个候选地点`;

// 创建 Agent（自动处理工具调用循环）
const poiPlannerAgent = createReactAgent({
  llm,
  tools: [searchPoiTool, getPlayerProfileTool, reverseGeocodeTool],
  prompt: POI_PLANNER_PROMPT,
});
```

**为什么用 createReactAgent：**

| 配置 | 理由 |
|------|------|
| `createReactAgent` | Agent 需要自主决定调用哪些工具、调用多少次 |
| `temperature: 0.3` | 任务明确，不需要太多随机性 |
| 工具 3 个 | 符合"3-5 个工具"原则 |
| 明确的 System Prompt | 对应 role + goal + backstory |

### 3.2 Agent 2: `storyGenerator`（纯 LLM 调用）

**职责**：基于候选 POI，生成完整的叙事路线

```typescript
import { ChatOpenAI } from "@langchain/openai";
import { z } from "zod";

// LLM 配置（支持 OpenAI 协议的模型）
const storyGenerator = new ChatOpenAI({
  model: "gpt-4o-mini",
  temperature: 0.7,  // 创意任务，稍高温度
  // 如需自定义 API 地址：
  // configuration: { baseURL: "https://your-api-endpoint/v1" },
});

const STORY_GENERATOR_PROMPT = `你是「今日人设」的故事引擎。

你的目标是：将真实地点编织成一段有情绪弧线的城市故事。

你的背景：
- 擅长用第二人称叙事
- 知道好的周末不是打卡清单，而是一段有起承转合的故事
- 文字克制但有温度，像朋友在耳边轻声建议
- 总是给具体的行动指引，而不是模糊的"感受氛围"

规则：
1. 从候选地点中选择 3-4 个，地理上能走通
2. 每个场景从人设的第一视角叙事
3. scene_name 是诗意命名（6-10字）
4. action_task 必须具体可执行
5. 整条路线要有情绪弧线`;

// 定义输出 Schema
const journeySchema = z.object({
  story_opening: z.string(),
  emotion_arc: z.object({ start: z.string(), end: z.string() }),
  scenes: z.array(z.object({
    order: z.number(),
    scene_name: z.string(),
    location_name: z.string(),
    location_type: z.string(),
    location_hint: z.string(),
    persona_narrative: z.string(),
    action_task: z.string(),
    stay_minutes: z.number(),
    emotion_tags: z.array(z.string()),
    meituan_keyword: z.string(),
  })).min(3).max(4),
  closing: z.string(),
});

// 绑定结构化输出
const storyGeneratorWithSchema = storyGenerator.withStructuredOutput(journeySchema);
```

**为什么不用 Agent：**

| 配置 | 理由 |
|------|------|
| 无工具 | 纯创作任务，不需要外部数据 |
| `withStructuredOutput` | 强制输出符合 Schema 的 JSON |
| `temperature: 0.7` | 创意任务，需要一定随机性 |
| 直接 LLM 调用 | 输入已由 `poiPlanner` 准备好，无需工具调用循环 |

## 4. Tools 设计（LangChain.js）

```typescript
import { tool } from "@langchain/core/tools";
import { z } from "zod";

// ===== Tool 1: 搜索 POI =====
const searchPoiTool = tool(
  async ({ keyword, radius = 3000 }) => {
    try {
      const url = `https://restapi.amap.com/v3/place/around?` +
        `key=${AMAP_KEY}&location=${lng},${lat}&` +
        `keywords=${encodeURIComponent(keyword)}&radius=${radius}&offset=5`;

      const res = await fetch(url).then(r => r.json());

      if (res.status !== "1") return [];

      return res.pois.slice(0, 5).map((p: any) => ({
        name: p.name,
        type: p.type?.split(";")[0] || "",
        address: p.address,
        location: p.location,
      }));
    } catch (e) {
      console.warn("search_poi failed:", keyword, e);
      return [];  // 失败返回空，不抛异常
    }
  },
  {
    name: "search_poi",
    description: "搜索附近的兴趣点。返回真实地点列表，包含名称、类型、地址。",
    schema: z.object({
      keyword: z.string().describe("搜索关键词，如：独立咖啡馆、小众公园、独立书店"),
      radius: z.number().optional().default(3000).describe("搜索半径（米）"),
    }),
  }
);

// ===== Tool 2: 获取玩家画像 =====
const getPlayerProfileTool = tool(
  async ({ playerKey }) => {
    const { data } = await supabase
      .from("dm_memory")
      .select("*")
      .eq("player_key", playerKey)
      .single();

    return data ?? {
      profile: "",
      loved_tags: [],
      disliked_tags: [],
      visited_pois: [],
    };
  },
  {
    name: "get_player_profile",
    description: "获取玩家的历史画像和偏好。包含：画像描述、喜欢的标签、不喜欢的标签、去过的地方。",
    schema: z.object({
      playerKey: z.string().describe("玩家唯一标识"),
    }),
  }
);

// ===== Tool 3: 逆地理编码 =====
const reverseGeocodeTool = tool(
  async ({ lat, lng }) => {
    // WGS84 → GCJ02 + 逆地理编码
    // ... 实现同现有代码
  },
  {
    name: "reverse_geocode",
    description: "将坐标转换为城市名称。返回：城市、区县、完整地址。",
    schema: z.object({
      lat: z.number().describe("纬度"),
      lng: z.number().describe("经度"),
    }),
  }
);
```

### 4.1 工具使用原则（参考 CrewAI）

| 原则 | 说明 |
|------|------|
| **工具数量** | 3-5 个/Agent，避免过多导致选择困难 |
| **工具描述** | 清晰说明用途，Agent 靠这个决定何时调用 |
| **输入 Schema** | 用 Zod 定义，提供字段描述 |
| **失败处理** | 工具内部 catch，返回空列表而非抛异常 |

## 5. LangGraph 流程设计

### 5.1 状态定义

```typescript
import { Annotation } from "@langchain/langgraph";

const QuestState = Annotation.Root({
  // 输入
  card: Annotation<PersonaCard>,
  city: Annotation<string>,
  lat: Annotation<number | undefined>,
  lng: Annotation<number | undefined>,
  playerKey: Annotation<string | undefined>,
  timePeriod: Annotation<string>,
  companion: Annotation<string>,
  
  // 中间状态
  playerProfile: Annotation<PlayerProfile | undefined>,
  poiCandidates: Annotation<POI[]>,
  
  // 输出
  journey: Annotation<Journey | undefined>,
});
```

### 5.2 节点实现

```typescript
import { StateGraph, START, END } from "@langchain/langgraph";

// ===== Node 1: 获取玩家画像 =====
async function fetchProfile(state: typeof QuestState.State) {
  if (!state.playerKey) return { playerProfile: undefined };

  const profile = await getPlayerProfile(state.playerKey);
  return { playerProfile: profile };
}

// ===== Node 2: POI 规划（使用 createReactAgent） =====
async function planPois(state: typeof QuestState.State) {
  const prompt = `
人设身份：${state.card.identity}
人设状态：${state.card.mood}
人设使命：${state.card.mission}
稀有度：${state.card.rarity}

城市：${state.city}
时间段：${state.timePeriod}

${state.playerProfile ? `
玩家画像：${state.playerProfile.profile}
喜欢的标签：${state.playerProfile.loved_tags.join("、")}
去过的地方：${state.playerProfile.visited_pois.slice(0, 10).join("、")}
` : ""}

请分析这个人设今天应该去什么类型的地点，搜索并收集 15-25 个候选。`;

  // 调用 createReactAgent（自动处理工具调用循环）
  const result = await poiPlannerAgent.invoke({
    messages: [{ role: "user", content: prompt }],
  });

  // 从结果中提取候选 POI
  const candidates = extractPOIsFromResult(result);
  return { poiCandidates: candidates };
}

// ===== Node 3: 生成路线（纯 LLM 调用） =====
async function generateJourney(state: typeof QuestState.State) {
  const prompt = `
人设身份：${state.card.identity}
人设状态：${state.card.mood}
人设使命：${state.card.mission}

候选地点（必须从中选择 3-4 个）：
${state.poiCandidates.map((p, i) => `${i+1}. ${p.name} | ${p.type} | ${p.address}`).join("\n")}

时间段：${state.timePeriod}
同伴：${state.companion}

请生成一条完整的城市剧情路线。`;

  // 直接调用 LLM（withStructuredOutput 强制输出 Schema）
  const journey = await storyGeneratorWithSchema.invoke([
    { role: "system", content: STORY_GENERATOR_PROMPT },
    { role: "user", content: prompt },
  ]);

  return { journey };
}
```

### 5.3 图构建

```typescript
import { StateGraph, START, END } from "@langchain/langgraph";

const workflow = new StateGraph(QuestState)
  // 添加节点
  .addNode("fetch_profile", fetchProfile)
  .addNode("plan_pois", planPois)
  .addNode("generate_journey", generateJourney)

  // 添加边
  .addEdge(START, "fetch_profile")
  .addEdge("fetch_profile", "plan_pois")
  .addEdge("plan_pois", "generate_journey")
  .addEdge("generate_journey", END);

const app = workflow.compile();
```
```

## 6. 待讨论问题

### 7.1 LLM 选择

| 选项 | 优点 | 缺点 |
|------|------|------|
| **GPT-4o-mini** | LangChain 原生支持、稳定、便宜 | 需要付费 |
| **国产模型（通义/DeepSeek）** | 支持 OpenAI 协议、便宜 | 需配置 baseURL |
| **本地模型（Ollama）** | 免费、隐私 | 需要本地部署 |

**已决定**：使用支持 OpenAI 协议的模型，通过 `ChatOpenAI` 配置 `baseURL` 即可切换。

```typescript
// 示例：使用国产模型
const llm = new ChatOpenAI({
  model: "qwen-plus",
  configuration: { baseURL: "https://dashscope.aliyuncs.com/compatible-mode/v1" },
});
```

### 7.2 工具调用模式

| Agent | 模式 | 说明 |
|------|------|------|
| `poiPlanner` | `createReactAgent` | Agent 自主决定调用哪些工具 |
| `storyGenerator` | 纯 LLM + `withStructuredOutput` | 无需工具，直接输出结构化结果 |

**已决定**：poiPlanner 用 createReactAgent，storyGenerator 用纯 LLM 调用。

### 7.3 延迟 vs 质量

| 配置 | 延迟 | 质量 |
|------|------|------|
| 单次调用（当前） | ~3s | 一般 |
| 固定流程 Agent | ~5-8s | 较好 |
| ReAct Agent | ~10-15s | 最好 |

**建议**：固定流程 + 条件重试
